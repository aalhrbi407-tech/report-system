# نظام متابعة وتقييم التزام الموظفين بالتقارير

تطبيق ويب متكامل (خادم Node + قاعدة بيانات Postgres + واجهة عربية RTL) يشغّل الآلية المؤتمتة:
لكل موظف رصيد تقارير مستهدف في الربع، ينخفض تلقائيًا **بعد اعتماد المشرف** لكل تقرير،
مع تقييم موزون (التزام زمني، اكتمال، جودة، كفاءة إغلاق)، وإدارة مستخدمين وصلاحيات وإحصائيات.

---

## النشر المجاني (Render + Neon) — بلا بطاقة ائتمان

المكوّنان مجانيان ودائمان ولا يطلبان بطاقة:
- **Neon** — قاعدة بيانات Postgres مجانية دائمة (تحفظ بياناتك).
- **Render** — استضافة الخادم على الخطة المجانية (تبقى منشورة، مع «إيقاظ» بعد فترة خمول).

> لماذا Postgres وليس ملفًا محليًا؟ لأن الخطة المجانية في Render لا تمنح قرصًا دائمًا،
> فلو خزّنّا البيانات محليًا ستُمسح عند كل إعادة تشغيل. لذلك تُحفظ البيانات في Neon.

### الخطوة 1 — جهّز قاعدة البيانات على Neon
1. أنشئ حسابًا على **neon.com** (بلا بطاقة).
2. أنشئ Project جديدًا (اختر أقرب منطقة لك).
3. من صفحة **Connection Details** انسخ **Connection string** (يبدأ بـ `postgres://...` وينتهي بـ `?sslmode=require`).
   احتفظ به؛ ستضعه في Render باسم `DATABASE_URL`.

### الخطوة 2 — ارفع الكود إلى GitHub
1. أنشئ مستودعًا جديدًا على GitHub.
2. ارفع محتويات هذا المجلد إليه (بدون مجلد `node_modules` — ملف `.gitignore` يتكفّل بذلك).

### الخطوة 3 — انشر على Render
1. أنشئ حسابًا على **render.com** (بلا بطاقة للخطة المجانية).
2. اضغط **New → Blueprint**، واختر مستودعك. سيقرأ Render ملف `render.yaml` تلقائيًا.
3. سيطلب منك تعبئة قيمتين سرّيتين:
   - `DATABASE_URL` = رابط Neon الذي نسخته.
   - `ADMIN_PASSWORD` = كلمة مرور قوية لمدير النظام.
   (أمّا `SESSION_SECRET` فيولّده Render تلقائيًا.)
4. اضغط **Apply / Create**. سيبني Render ويشغّل النظام ويعطيك رابطًا عامًا مع HTTPS.

> بديل يدوي بدون Blueprint: New → Web Service → اربط المستودع →
> Build: `npm install` — Start: `npm start` — ثم أضف متغيّرات البيئة يدويًا
> (`DATABASE_URL`, `SESSION_SECRET`, `SECURE_COOKIE=1`, `ADMIN_PASSWORD`).

### الخطوة 4 — أول دخول
افتح الرابط، وسجّل الدخول بـ `admin` وكلمة المرور التي وضعتها، ثم **غيّرها فورًا** من «الإعدادات».
بعدها: أنشئ المستخدمين، وحدّد الصلاحيات، وأسند المستهدفات — وابدأ التشغيل.

### ملاحظات على الخطة المجانية
- **الإيقاظ (Cold start):** خدمة Render المجانية «تنام» بعد ~15 دقيقة خمول، فيتأخّر أول طلب ~30–60 ثانية ثم تعمل بشكل طبيعي. مناسب لأداة داخلية.
- **قاعدة Neon المجانية:** 0.5GB (تكفي عشرات آلاف التقارير) وتنام عند الخمول وتستيقظ خلال أجزاء من الثانية.
- للتشغيل بلا انتظار إيقاظ، خطة Render المدفوعة (~7$ شهريًا) تلغي النوم.

## التشغيل محليًا (للتجربة قبل النشر)

```bash
npm install
cp .env.example .env      # يكفي تركه كما هو للتجربة المحلية
npm start                 # ثم افتح http://localhost:3000
```

محليًا وبدون `DATABASE_URL` يستخدم النظام قاعدة **PGlite** مدمجة (Postgres داخل العملية)
وتُحفظ في مجلد `data/pg` — بلا أي إعداد خارجي. سجّل الدخول بحساب المدير من `.env`.

لمعاينة النظام ببيانات فريق تجريبية: `npm run seed:demo` (كلمات المرور `pass123`، المشرفة `sara`).

## الأمان — قبل إدخال بيانات موظفين حقيقيين

النظام مبنيّ على أسس سليمة: كلمات مرور مشفّرة بـ bcrypt، جلسات موقّعة في كوكيز httpOnly،
وفحص كل صلاحية على الخادم لكل طلب (لا يكفي إخفاء الأزرار في الواجهة). ومع ذلك أنصح قبل الإطلاق الرسمي:

- **HTTPS إلزامي** مع `SECURE_COOKIE=1` (يوفّره Render تلقائيًا).
- **غيّر كلمة مرور المدير** فور أول دخول، واحفظ `DATABASE_URL` و`SESSION_SECRET` سرًّا.
- **النسخ الاحتياطي:** فعّل النسخ التلقائي في Neon، أو صدّر نسخة دورية.
- **حدّ محاولات الدخول (rate limiting):** غير مضمّن افتراضيًا؛ يُنصح بإضافته ضد التخمين.
- **رفع الملفات:** يُسجَّل «اسم» المرفق فقط حاليًا ولا يُخزَّن الملف فعليًا؛ الرفع الحقيقي ميزة تحتاج إضافة.
- **الخصوصية والامتثال:** بما أن النظام يعالج بيانات موظفين، راجِع فريق تقنية المعلومات والأمن لديك
  بشأن مكان الاستضافة ومتطلبات حماية البيانات (مثل نظام حماية البيانات الشخصية السعودي PDPL)
  قبل الإطلاق. هذه إرشادات عامة وليست استشارة قانونية.

## أوامر مفيدة

| الأمر | الوظيفة |
|------|---------|
| `npm start` | تشغيل الخادم |
| `npm run seed:demo` | تحميل بيانات فريق تجريبية (معاينة فقط) |
| `npm run reset` | مسح كل البيانات وإعادة إنشاء حساب المدير فقط ⚠️ |
| `node test-e2e.js` | تشغيل اختبارات النظام الآلية (يستخدم PGlite داخليًا) |

## بنية المشروع

```
report-system/
├── server.js        # خادم Express: المصادقة، الـ API، فحص الصلاحيات
├── db.js            # طبقة Postgres: المخطّط، المهيّئ، لقطة الحالة (Neon أو PGlite)
├── seed-cli.js      # أدوات التهيئة والمسح من سطر الأوامر
├── test-e2e.js      # اختبارات آلية شاملة (38 اختبارًا)
├── public/
│   └── index.html   # الواجهة العربية (RTL) كاملة
├── render.yaml      # مخطّط النشر التلقائي على Render
├── .env.example     # قالب متغيّرات البيئة
└── package.json
```

## نموذج التقييم

الدرجة الكلية لكل تقرير معتمد =
`الالتزام الزمني×وزنه + الاكتمال×وزنه + الجودة×وزنها + كفاءة الإغلاق×وزنها`.
الالتزام الزمني وكفاءة الإغلاق يُحتسبان آليًا؛ والاكتمال والجودة يقيّمهما المشرف عند الاعتماد.
الأوزان قابلة للتعديل من «الإعدادات» ومجموعها 100%.
